# 📐 เอกสารออกแบบระบบ — Climate & Geo-Risk Monitor

> เวอร์ชัน 1.0 · ขอบเขต: ไทย (หลัก) + เอเชียตะวันออกเฉียงใต้ + ทั่วโลก · ติดตามแบบหลายจุด (multi-point)
> ภาคส่วน: เกษตร/อาหาร/น้ำ · การเงิน/ลงทุน/ประกัน · โลจิสติกส์/ซัพพลายเชน · อสังหา/ก่อสร้าง/อีเวนต์ · พลังงาน · สุขภาพ/ครัวเรือน

## 1. เป้าหมายและขอบเขต

**ปัญหา:** โปรเจกต์ของผู้ใช้ผูกกับสถานการณ์โลกอย่างใกล้ชิด ต้องรู้ล่วงหน้าว่า *ช่วงไหน* ต้อง
ระวัง *เรื่องอะไร* ที่ *พื้นที่ใด* เพื่อเตรียมตัว ไม่ใช่ตามข่าวไวรัลที่ปนข้อมูลผิด

**สิ่งที่ระบบเป็น:**
- ตัวรวบรวมพยากรณ์/ข้อมูลจากสถาบันทางการหลายแห่ง → ทำให้อ่านง่ายในที่เดียว
- ตัวแปลงข้อมูลภัย → "คะแนนความเสี่ยงเฉพาะโปรเจกต์/พื้นที่/ภาคส่วน" ของผู้ใช้
- ตัวแจ้งเตือนล่วงหน้า + ออกบรีฟเตรียมตัวอัตโนมัติ
- ตัว fact-check ข่าว/โพสต์ไวรัลเทียบกับแหล่งทางการ

**สิ่งที่ระบบไม่เป็น (ขอบเขตที่ตัดออกโดยตั้งใจ):**
- ❌ ไม่สร้างแบบจำลองพยากรณ์อากาศเชิงตัวเลข (NWP) แข่งกับ NOAA/ECMWF — ไม่คุ้มและไม่แม่นกว่า
- ❌ ไม่รับประกันคำพยากรณ์ — เป็นเครื่องมือ "เตรียมความพร้อม" ไม่ใช่คำทำนาย
- ❌ ไม่ใช่ระบบเตือนภัยฉุกเฉินทางการ (ต้องอ้างอิงประกาศรัฐเสมอ)

## 2. หลักการออกแบบ

| หลักการ | ความหมายเชิงปฏิบัติ |
|---|---|
| **Aggregate, don't reinvent** | ดึงพยากรณ์ทางการมาใช้ ไม่คำนวณฟิสิกส์เอง |
| **Source of truth คือสถาบัน** | ทุกตัวเลขมี citation + เวลาดึง + ลิงก์ต้นทาง |
| **Graceful degradation** | ดึงไม่ได้ → ใช้ค่าล่าสุด/ค่าสำรอง ระบบไม่ล่ม |
| **Project-centric** | ความเสี่ยงคำนวณต่อ "โปรเจกต์ + พิกัด + ภาคส่วน" ไม่ใช่ภาพรวมลอยๆ |
| **Explainable risk** | ทุกคะแนนเสี่ยงบอกเหตุผล (hazard อะไร, ข้อมูลไหน) |
| **Anti-misinformation** | แยก "ข้อเท็จจริง" ออกจาก "การตีความ/กลไก" เสมอ |

## 3. ผู้ใช้และ Use Cases (ตามภาคส่วนที่เลือก)

| ภาคส่วน | คำถามที่ระบบต้องตอบ | ภัยหลักที่เกี่ยว |
|---|---|---|
| **เกษตร/อาหาร/น้ำ** | ควรเลื่อนเพาะปลูกไหม? น้ำต้นทุนพอถึงเมื่อไหร่? | แล้ง, ฝนทิ้งช่วง, น้ำท่วม, คลื่นความร้อน |
| **การเงิน/ลงทุน/ประกัน** | ราคาสินค้าเกษตร/พลังงานจะตึงช่วงไหน? ความเสี่ยงเคลมประกันพุ่งเมื่อไหร่? | แล้ง, น้ำท่วม, พายุ (กระทบราคา/เคลม) |
| **โลจิสติกส์/ซัพพลายเชน** | เส้นทาง/ท่าเรือไหนเสี่ยงสะดุดช่วงพายุ? | ไต้ฝุ่น, น้ำท่วม, คลื่นความร้อน |
| **อสังหา/ก่อสร้าง/อีเวนต์** | หน้างานเสี่ยงหยุดช่วงไหน? อีเวนต์กลางแจ้งวันไหนเสี่ยงร้อน/ฝน/พายุ? | คลื่นความร้อน, ฝน/พายุ, น้ำท่วม, แผ่นดินไหว (โครงสร้าง) |
| **พลังงาน** | พีคโหลด/ไฟดับเสี่ยงช่วงไหน? พลังน้ำลดเมื่อไหร่? | คลื่นความร้อน, แล้ง (เขื่อน) |
| **สุขภาพ/ครัวเรือน** | วันไหนต้องเลี่ยงแดด/ฝุ่น? | คลื่นความร้อน, PM2.5, โรคหลังน้ำท่วม |

## 4. โมเดลพื้นที่ (Geo Model) — หลายจุด, 3 ระดับ

ผู้ใช้นิยาม **"จุดที่ต้องติดตาม" (watch points)** ได้หลายจุด แต่ละจุดผูกกับโปรเจกต์/ภาคส่วน:

```
Watch Point = { name, lat, lon, radius_km, sectors[], projectId }
```

ระดับการติดตาม 3 ชั้นซ้อนกัน:
1. **Global signals** — ENSO, อุณหภูมิโลก, แผ่นดินไหวสำคัญ (บริบทภาพใหญ่)
2. **Regional (SEA)** — มรสุม, เส้นทางไต้ฝุ่น, SST แปซิฟิก/ทะเลจีนใต้ (กระทบทั้งภูมิภาค)
3. **Local (จุดของผู้ใช้)** — heat index, ฝน, น้ำท่วม, PM2.5, ระดับน้ำเขื่อนใกล้จุด

> การ map ภัยลงแต่ละจุด: ใช้พิกัด query กริดข้อมูล (เช่น Open-Meteo รับ lat/lon ตรง) +
> ผูกจุดกับลุ่มน้ำ/เขื่อน/สถานีตรวจที่ใกล้ที่สุด

## 6. แหล่งข้อมูล (Data Sources)

### 6.1 ระดับโลก/ภูมิภาค
| ภัย/ตัวชี้วัด | แหล่ง | รูปแบบ | ความถี่ | คีย์ |
|---|---|---|---|---|
| ENSO / ONI | NOAA CPC | text ascii | เดือน/สัปดาห์ | ไม่ต้อง |
| ENSO forecast | IRI Columbia | web/CSV | เดือน | ไม่ต้อง |
| SST anomaly | NOAA OISST / Coral Reef Watch | NetCDF/PNG | วัน | ไม่ต้อง |
| แผ่นดินไหว | USGS, EMSC | GeoJSON | เรียลไทม์ | ไม่ต้อง |
| พายุหมุนเขตร้อน | JTWC, JMA | web/feed | 6 ชม. | ไม่ต้อง |
| สภาพอากาศจุด (lat/lon) | **Open-Meteo** | JSON | ราย ชม. | ไม่ต้อง (ฟรี) |
| คุณภาพอากาศ/PM2.5 | Open-Meteo Air Quality, OpenAQ | JSON | ราย ชม. | ฟรี/คีย์ |

### 6.2 ระดับไทย
| ภัย/ตัวชี้วัด | แหล่ง | หมายเหตุ |
|---|---|---|
| อากาศ/พายุ/เตือนภัย | กรมอุตุนิยมวิทยา (TMD) | มี API/หน้าเตือนภัย |
| น้ำ/เขื่อน/แล้ง | สทนช., กรมชลประทาน, คลังข้อมูลน้ำ (NHC/Thaiwater) | ระดับน้ำเขื่อนรายวัน |
| น้ำท่วม | GISTDA, ปภ. (DDPM) | แผนที่น้ำท่วม/ประกาศเขตภัย |
| PM2.5 | Air4Thai (กรมควบคุมมลพิษ) | สถานีรายชั่วโมง |

> **กลยุทธ์การต่อแหล่งไทย:** เริ่มจากแหล่งที่มี API/ไฟล์เปิด (Thaiwater, Air4Thai) ก่อน;
> แหล่งที่ไม่มี API ใช้ scraping เบาๆ + cache และมี Verification Agent ตรวจความสอดคล้อง

### 6.3 นโยบายข้อมูล
- เก็บ raw payload ลง `sources_audit` ทุกครั้ง (ตรวจย้อนหลัง/ debug ได้)
- บันทึก license/ที่มาทุกแหล่ง; แสดง citation ใน UI เสมอ
- เคารพ rate limit + robots.txt; cache ตามความถี่จริงของข้อมูล

## 9. วิธีคิดคะแนนความเสี่ยง (Risk Scoring) — แกนหลักของระบบ

ใช้กรอบมาตรฐาน UNDRR/IPCC:

```
Risk = Hazard × Exposure × Vulnerability
```

- **Hazard (H):** ความรุนแรง/โอกาสเกิดภัย จากข้อมูล/พยากรณ์ (normalize 0–1)
  เช่น แล้ง → จาก ONI + ระดับน้ำเขื่อน + SPI; ร้อน → จาก heat index พยากรณ์
- **Exposure (E):** จุด/โปรเจกต์อยู่ในพื้นที่รับภัยแค่ไหน (0–1) จากพิกัด + ลุ่มน้ำ
- **Vulnerability (V):** ภาคส่วนนั้นอ่อนไหวต่อภัยนี้แค่ไหน (เมทริกซ์น้ำหนักตายตัว เช่น
  เกษตร × แล้ง = สูง, การเงิน × แผ่นดินไหว = ต่ำ)

**ผลลัพธ์:** คะแนน 0–100 → จัดระดับ `low / moderate / high / severe` ต่อ (จุด × ภาคส่วน × ช่วงเวลา)
พร้อม `rationale` อธิบายว่ามาจาก hazard ใดและข้อมูลไหน

ตัวอย่างเมทริกซ์ Vulnerability (V) — น้ำหนัก 0–1:

| ภาคส่วน \ ภัย | แล้ง | ร้อน | น้ำท่วม | พายุ | แผ่นดินไหว | PM2.5 |
|---|---|---|---|---|---|---|
| เกษตร/น้ำ | 1.0 | 0.8 | 0.7 | 0.6 | 0.1 | 0.2 |
| การเงิน/ประกัน | 0.6 | 0.4 | 0.8 | 0.8 | 0.7 | 0.2 |
| โลจิสติกส์ | 0.3 | 0.5 | 0.9 | 0.9 | 0.5 | 0.2 |
| อสังหา/ก่อสร้าง/อีเวนต์ | 0.2 | 0.8 | 0.8 | 0.8 | 0.7 | 0.5 |
| พลังงาน | 0.7 | 0.9 | 0.5 | 0.6 | 0.3 | 0.1 |
| สุขภาพ/ครัวเรือน | 0.3 | 1.0 | 0.6 | 0.5 | 0.4 | 1.0 |

## 10. Agents / Skills

| Agent | หน้าที่ | Trigger | เครื่องมือ/MCP |
|---|---|---|---|
| **Ingestion** | ดึง+ทำความสะอาด+เก็บลง DB | cron ตามความถี่แต่ละแหล่ง | fetch, Supabase |
| **Verification** | ตรวจข่าว/โพสต์เทียบแหล่งทางการ → ให้คะแนนจริง/บางส่วน/ผิด | on-demand / เมื่อมีโพสต์เข้า | WebSearch, WebFetch |
| **Risk-Scoring** | คำนวณ H×E×V ต่อจุด×ภาคส่วน×ช่วง → เขียน `risk_scores` | หลัง ingestion / รายวัน | Supabase, สูตรในข้อ 9 |
| **Briefing** | สรุปบรีฟราย สัปดาห์/เดือน "ช่วงไหนระวังอะไรที่จุดใด" | cron รายสัปดาห์ | LLM + Supabase |
| **Alert** | threshold ทะลุ → ส่งเตือน + log | event-driven | LINE/Email/Push |
| **Orchestrator** | จัดคิว/ตารางเวลา รวมผล จัดการ error/retry | ตลอดเวลา | cron + ทุก agent |

> **ทำไมต้องเป็น "agent" ไม่ใช่แค่ cron script:** Verification และ Briefing ต้องใช้
> การให้เหตุผลภาษาธรรมชาติ (อ่านโพสต์ เทียบหลายแหล่ง สรุปเชิงบริบท) — ส่วน Ingestion/
> Risk-Scoring เป็น deterministic job ได้ (ไม่ต้องใช้ LLM ทุกครั้ง = ประหยัด)

## 11. Dashboard / UX

หน้า/คอมโพเนนต์:
1. **Overview** — สถานะ ENSO, ภัยเด่นวันนี้, คะแนนเสี่ยงรวมของทุกจุด
2. **Map หลายจุด** — หมุดแต่ละ watch point ระบายสีตามระดับเสี่ยงรวม
3. **ปฏิทินเตรียมตัว 12 เดือน** — (มีแล้วใน MVP) กรองตามภาคส่วน/จุด
4. **Risk Matrix** — ตาราง จุด × ภาคส่วน ระบายสี + คลิกดู rationale
5. **Hazard panels** — ENSO timeline, earthquake feed, typhoon tracker, AQI, ระดับน้ำเขื่อน
6. **Alerts log** — ประวัติการเตือน + สถานะ
7. **Fact-check** — (มีแล้ว) + ช่องวางโพสต์ใหม่ให้ Verification Agent ตรวจ
8. **Briefing export** — ออกบรีฟเป็น PDF/ภาพ/ข้อความ LINE

## 13. การแจ้งเตือน (Alerting)

- ช่องทาง: **LINE Notify / Email / In-app push** (เริ่มที่ LINE เพราะคนไทยใช้มากสุด)
- เกณฑ์: เมื่อ `risk_score` ของจุดใดข้ามขั้น (เช่น high→severe) หรือมี event ใหญ่ใกล้จุด
  (แผ่นดินไหว ≥ M5 ในรัศมี, พายุเข้าเส้นทาง, heat index > เกณฑ์)
- กันสแปม: dedupe + cooldown ต่อจุด/ภัย; รวมหลายเตือนเป็นบรีฟเดียวถ้าถี่เกิน

- ทุกข้อมูลภายนอกถือเป็น untrusted → validate ก่อนใช้ (กัน prompt injection ใน Verification)

## 17. ข้อจำกัดและความเสี่ยงของระบบ

- คุณภาพผลลัพธ์ = คุณภาพแหล่งข้อมูล; พยากรณ์ตามฤดูกาลมีความไม่แน่นอนสูงโดยธรรมชาติ
- แหล่งไทยบางส่วนไม่มี API ทางการ → ต้อง scrape (เปราะ ต้องมี monitoring)
- Risk score เป็น "ดัชนีช่วยตัดสินใจ" ไม่ใช่ความจริงสัมบูรณ์ — ต้องโชว์ rationale + ที่มาเสมอ
- ต้องกัน misinformation: แยกข้อเท็จจริงจากการตีความ และเตือนเมื่อข้อมูลขัดแย้งกัน
